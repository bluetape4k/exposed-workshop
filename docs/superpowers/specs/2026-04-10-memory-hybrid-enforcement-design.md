# Memory Hybrid Enforcement Design

## 개요

이 설계의 목적은 Claude Code가 이 프로젝트에서 자주 놓치는 사용자 선호와 작업 방식을 줄이는 것이다. 특히 우선순위는 다음과 같다.

1. 테스트 코드 작성 및 실행 누락 방지
2. 코딩 스타일과 응답 방식 준수 강화
3. 기존 mempalace, wiki, obsidian 자산의 역할을 재정의하여 중복 저장은 줄이고 회수율은 높이기

핵심 방향은 hybrid 접근이다. 테스트 관련 규칙은 hook으로 강하게 강제하고, 코딩 스타일과 응답 습관은 auto-memory와 경량 reminder로 유지한다. mempalace, wiki, obsidian은 검색/아카이브 자산으로 유지하되, 행동 강제의 1차 장치로 사용하지 않는다.

## 문제 정의

현재 forgetting 체감의 핵심은 저장 부족이 아니라 retrieval과 execution enforcement 부족이다.

- auto-memory에는 일부 규칙이 저장되어 있으나, 실행 직전 행동을 막는 강제력이 약하다.
- mempalace는 chats markdown을 mining 하여 Chroma index를 구축하지만, 매 턴 자동 주입되는 행동 규칙 저장소가 아니다.
- wiki/obsidian은 재사용 가치 있는 문서에는 적합하지만, 테스트 누락 방지 같은 즉시성 높은 규칙을 강제하기엔 느슨하다.
- 그 결과 장기적으로는 많이 저장되지만, 실제 구현 순간에는 테스트 작성·실행·기록이 빠질 수 있다.

## 목표

### 1. 테스트 우선 행동 강제

코드 파일을 수정한 뒤 다음을 빠뜨리지 않도록 만든다.

- 관련 테스트 작성 또는 기존 테스트 근거 확인
- 테스트 실행
- 테스트 로그 기록 (`docs/testlogs/YYYY-MM.md`)

### 2. 코딩 스타일 준수율 향상

다음 규칙을 세션 시작과 사용자 프롬프트 시점에 짧고 명확하게 상기시킨다.

- 확인 후 답변, 추측 금지
- 요청 범위 밖 확장 금지
- 변경 후 테스트 필수
- 모듈 변경 시 README 동기화

### 3. 지식 저장소 역할 분리

각 저장소를 아래처럼 분리해 혼선을 줄인다.

- auto-memory: 행동 규칙, 사용자 선호, 자주 반복되는 협업 규칙
- mempalace: 장기 검색용 환경 지식, 도구 설정, 회고, 세션 요약
- wiki: 반복 사용 가능한 패턴, 절차, 결정사항 문서
- obsidian: 개인 지식 정리 및 장문 문서 연결 허브

## 접근 방식 비교

### 접근 1. Memory-first

모든 문제를 auto-memory 정리와 요약 강화로 해결하려는 방식이다.

장점:
- 구현이 간단하다.
- 사용자 경험이 덜 거슬린다.

단점:
- 테스트 누락 방지력이 약하다.
- 기억은 있어도 실행 순간 빠질 수 있다.

### 접근 2. Hook-first

모든 forgetting 문제를 hook 차단으로 해결하려는 방식이다.

장점:
- 강제력이 가장 높다.
- 테스트 누락을 즉시 줄인다.

단점:
- 과도하게 엄격하면 작업 흐름이 답답해질 수 있다.
- 코딩 스타일 같은 softer rule까지 차단으로 다루면 피로도가 커진다.

### 접근 3. Hybrid (채택)

테스트는 hook으로 강제하고, 코딩 스타일은 memory + reminder로 유지한다. mempalace/obsidian은 검색/문서화 채널로 재배치한다.

장점:
- 현재 우선순위(테스트 누락 > 스타일)를 정확히 반영한다.
- 강제력과 유연성의 균형이 좋다.
- 기존 저장 자산을 버리지 않고 역할만 명확히 나눈다.

단점:
- 설정 포인트가 2~3곳으로 나뉜다.
- 초기 튜닝이 필요하다.

## 아키텍처

### A. 테스트 강제 레이어

`PostToolUse(Edit|Write)` 계층에서 코드 수정 이벤트를 감지한다.

검사 대상:
- Kotlin/Gradle 관련 코드 파일 변경
- 테스트 파일이 아닌 구현 파일 변경

검사 논리:
1. 변경 파일이 테스트 대상인지 판별한다.
2. 관련 테스트 파일 변경 또는 존재 여부를 본다.
3. 최근 Bash 실행 내역에서 관련 테스트 실행 흔적을 본다.
4. 같은 작업 흐름에서 `docs/testlogs/YYYY-MM.md` 기록 여부를 본다.

결과 처리:
- 1차: 강한 경고와 구체적 누락 항목 출력
- 반복 누락 시: Stop hook에서 미해결 항목 재점검
- 필요 시 후속 단계에서 block 모드로 강화 가능

### B. 행동 규칙 회상 레이어

auto-memory와 session reminder를 간결하게 유지한다.

유지할 핵심 규칙 예시:
- 확인 후 답변, 추측 금지
- 테스트 코드/실행/로그 누락 금지
- 모듈 변경 시 README/README.ko 동기화
- 요청 범위 밖 확장 금지

이 규칙은 길게 저장하지 않고, 다음 세션에서도 즉시 행동으로 이어질 수 있게 짧고 명령적으로 유지한다.

### C. 검색 및 문서 자산 레이어

#### mempalace

다음 유형만 저장 대상으로 강화한다.
- 환경 설정 절차
- MCP/도구 설치 정보
- 장기적으로 반복되는 디버깅 결과
- 프로젝트 전반 의사결정 요약

반대로 다음은 mempalace의 1차 저장 대상으로 두지 않는다.
- 세션 중 즉시 강제해야 하는 행동 규칙
- 짧은 협업 선호 규칙

#### wiki

다음 유형을 저장한다.
- 반복 사용하는 절차
- 아키텍처 결정
- 삽질해서 알아낸 디버깅 패턴
- 팀 차원에서 공유할 가치가 있는 설정 방법

#### obsidian

다음 역할로 제한한다.
- 개인 지식 베이스 허브
- wiki/mempalace/chat 요약을 연결하는 장문 노트
- 개념 지도 및 링크드 노트

obsidian은 Claude의 강제 규칙 저장소가 아니라, 사람이 탐색하는 지식 공간으로 둔다.

## 컴포넌트 설계

### 1. test-enforcement hook

역할:
- 코드 변경 직후 테스트 관련 누락을 탐지한다.

입력:
- `tool_input.file_path`
- 최근 작업 파일 컨텍스트
- Bash 실행 흔적

출력:
- 누락 항목 목록
- 권장 다음 액션

검사 항목:
- 테스트 파일 작성/수정 여부
- 테스트 실행 여부
- testlog 기록 여부
- 문서 동기화 여부(모듈 변경 시)

### 2. session-priority reminder

역할:
- 세션 시작/프롬프트 제출 시 최우선 협업 규칙을 짧게 주입한다.

출력 예시:
- 테스트 우선: 코드 변경 후 테스트/실행/testlog 기록
- 확인 우선: 실제 확인 후 응답
- 과도한 확장 금지

### 3. mempalace export classifier

역할:
- Stop 시점에 저장할 내용을 분류한다.

분류 기준:
- 행동 규칙 → auto-memory
- 장기 검색 가치 있는 환경/설정/회고 → mempalace chats
- 반복 재사용 가치가 큰 것 → wiki
- 장문 연결 노트 필요 → obsidian

중복 저장 방침:
- 같은 내용을 모든 저장소에 복제하지 않는다.
- 원문 1개 + 포인터/요약 구조를 기본으로 한다.

## 데이터 흐름

### 코드 변경 흐름

1. Claude가 코드 파일 수정
2. `PostToolUse(Edit|Write)` 실행
3. test-enforcement가 관련 테스트/실행/log 여부 검사
4. 누락 시 즉시 경고 출력
5. 사용자가 테스트 보완 후 진행
6. Stop 시점에서 미해결이면 재알림

### 지식 저장 흐름

1. 세션 종료 시 중요한 산출물과 결정사항 정리
2. 행동 규칙은 memory에만 반영
3. 장기 참고 지식은 mempalace chat markdown으로 저장
4. 재사용 가치가 높으면 wiki 문서 생성 또는 갱신
5. obsidian은 선택적으로 링크/요약 허브 갱신

## 에러 처리

### Hook 오탐

테스트가 필요 없는 변경(예: doc-only, comment-only, 명백한 비코드 변경)은 경고를 생략한다. 이를 위해 파일 확장자와 경로 패턴으로 1차 필터링한다.

### Hook 누락

테스트는 했지만 탐지에 실패한 경우를 줄이기 위해, 테스트 실행 흔적은 한 가지 방식만 보지 않고 다음 중 하나라도 만족하면 통과로 본다.

- Gradle test task 실행 흔적
- repo-test-summary 실행 흔적
- testlogs 기록 존재

### 저장소 중복

동일 지식이 mempalace/wiki/obsidian에 중복 저장되면 관리 비용이 커진다. 분류기에서 저장 목적을 먼저 판정하고, 포인터 기반 저장을 우선한다.

## 테스트 전략

### 검증 시나리오 1 — 구현 파일만 수정

- 구현 Kotlin 파일 수정
- 테스트 파일 미수정
- 테스트 미실행
- 기대 결과: hook이 테스트 누락 경고

### 검증 시나리오 2 — 구현 + 테스트 + testlog 완료

- 구현 파일 수정
- 테스트 파일 수정
- 관련 Gradle test 실행
- `docs/testlogs/YYYY-MM.md` 기록
- 기대 결과: hook 경고 없음

### 검증 시나리오 3 — doc-only 변경

- README 또는 markdown만 수정
- 기대 결과: 테스트 경고 없음

### 검증 시나리오 4 — session 종료 저장

- 환경 설정/도구 설치 관련 세션 종료
- 기대 결과: mempalace chat 저장 또는 wiki 저장
- 행동 규칙은 memory 쪽으로만 반영

## 구현 범위

이번 단계에서 다룰 범위:
- 테스트 강제 hook 설계 및 적용
- memory 규칙 압축 정리
- mempalace/wiki/obsidian 역할 분리 및 연동 정리

이번 단계에서 제외:
- mempalace 자체 schema 변경
- obsidian 대규모 vault 구조 개편
- 외부 서비스 연동 추가

## 성공 기준

- 코드 수정 후 테스트 누락 경고가 일관되게 발생한다.
- 세션 시작 시 최우선 행동 규칙이 짧고 명확하게 상기된다.
- mempalace, wiki, obsidian의 역할이 분리되어 중복 저장이 줄어든다.
- 사용자가 체감하는 forgetting, 특히 테스트 누락이 현저히 감소한다.

## 결정사항

- 하이브리드 접근을 채택한다.
- 테스트 누락 방지가 최우선이다.
- 코딩 스타일은 차단보다 회상과 상기로 다룬다.
- mempalace/obsidian은 강제 장치가 아니라 검색/지식 자산으로 사용한다.
